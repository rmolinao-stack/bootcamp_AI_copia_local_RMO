¿Qué es un "proveedor de IA"?

Cuando construís un asistente con un LLM, no programáis el modelo desde cero. Contratáis o accedéis a un servicio que expone ese modelo a través de una API: le mandáis texto, os devuelve texto.

Tu código
→
Petición a la API
(mensajes + parámetros)
→
Servidor del proveedor
(el LLM vive ahí)
→
Respuesta
(texto generado)

En el mercado coexisten muchos actores, cada uno con modelos, precios, límites y políticas de privacidad distintos:

TipoQué esEjemplo
Proveedor de modelo + APIEmpresa que entrena o distribuye LLMs y los sirve en la nubeGoogle Gemini, OpenAI, Cohere, Anthropic
Hub + inferenciaCatálogo de modelos open source y APIs para ejecutarlosHugging Face
Cloud hyperscalerIA integrada en la nube de AWS, Azure o Google CloudAmazon Bedrock, Azure OpenAI, Vertex AI
Hosting de modelos abiertosInfraestructura que ejecuta Llama, Mistral, etc.Groq, Together AI, Fireworks
No hay un único "mejor" proveedor: depende de coste, latencia, calidad en vuestro dominio, cumplimiento (dónde viven los datos) y de si ya usáis un ecosistema concreto.

Hoy vamos a mirar de cerca dos caminos muy distintos para acceder a un LLM: una API de pago/capa gratuita en la nube (Cohere) y la ejecución en tu propia máquina (Hugging Face local) — con una parada intermedia para entender bien la plataforma de Hugging Face.

1

Cohere — Chat API en la nube

Un proveedor "clásico": te registras, pides una clave, llamas a un endpoint.

¿Qué es Cohere?

Cohere es un proveedor de modelos de lenguaje orientado a aplicaciones empresariales: chat, clasificación, extracción y embeddings. Ofrece API en la nube con una capa gratuita de prueba — ideal para aprender y prototipar, como hoy.

Cómo funciona una llamada de chat

messages = [ {"role": "system", "content": "..."} ← instrucciones fijas del asistente {"role": "user", "content": "..."} ← lo que pregunta la persona {"role": "assistant", "content": "..."} ← turnos anteriores del modelo ] client.chat(model="command-r-plus", messages=messages, temperature=0.2)

🎭 Los tres roles

system: quién es el asistente y sus límites (se manda una vez, fijo).
user: lo que escribe la persona.
assistant: respuestas previas del modelo, para dar memoria a la conversación.

🌡️ El parámetro temperature

Controla cuánto "se arriesga" el modelo al elegir palabras. Valores bajos → respuestas más predecibles y repetibles. Valores altos → más variadas y creativas (y más impredecibles).

Prueba tú mismo — ¿qué significa la temperature?

Mueve el control y observa cómo cambiaría el comportamiento del modelo (esto es una simulación visual, no llama a ninguna API):

0.0 — Predecible0.51.0 — Creativo
temperature = 0.2 → Ideal para asistentes que deben ser consistentes y fiables (onboarding, soporte, datos).

Cómo obtener la API key

1

Regístrate

En dashboard.cohere.com — cuenta gratuita.

2

Ve a "API Keys"

En el menú del dashboard, crea una clave de prueba (Trial key).

3

Cópiala y guárdala

La pegaréis en el notebook cuando os la pida (input oculto, no se ve al escribir).

Nunca subáis una API key a un repositorio. Se pide con getpass en el notebook y no se guarda en el código.

Tour por Hugging Face

Antes de tocar código: entender la plataforma. Es la pieza que más confunde, así que vamos despacio.

Hugging Face no es "un modelo": es un hub, un catálogo enorme de modelos open source, más un conjunto de servicios para ejecutarlos.

Inference Providers es el aeropuerto entero: todas las puertas, todas las aerolíneas. provider="hf-inference" es una única terminal, pequeña, con pocos vuelos. Si no indicáis provider=, el router os lleva a la puerta correcta automáticamente — eso es lo que queremos para chat.
Tu notebook → InferenceClient (router) → proveedor serverless
(Groq, Together, Cerebras...) → texto generado

Glosario exprés

Haz clic en cada término para ver la explicación:

Hub +

El catálogo de modelos de HF (huggingface.co/models). No es un proveedor en sí — es como el "GitHub de los modelos".

Inference Providers +

El sistema de HF que reenvía tu petición a un proveedor externo (Groq, Together, Cerebras...) que ejecuta el modelo por ti.

Router automático +

Lo que usaremos en el notebook: InferenceClient(api_key=...) sin más. HF elige el proveedor por vosotros.

provider="hf-inference" +

Un backend concreto y limitado de HF (no el router). Catálogo pequeño, sin chat con Llama. En este bootcamp: solo para embeddings.

Modelo gated +

Modelo que exige aceptar una licencia en la web de HF antes de poder descargarlo o usarlo.

Caché local +

Carpeta (~/.cache/huggingface/hub/) donde se guardan los pesos ya descargados, para no volver a descargarlos cada vez.

¿Dónde encontrar un modelo?

1

Catálogo oficial de Inference Providers

huggingface.co/inference/models — solo aparecen modelos llamables por API. Copiad el id (p. ej. openai/gpt-oss-20b).

2

La ficha del modelo

En cualquier repo del Hub, pestaña Deploy → Inference Providers: comprueba si hay proveedor disponible.

3

Ojo con lo que no está en el catálogo

Si un modelo no está en el catálogo de inferencia, la llamada de chat puede fallar directamente.

🎯 Misión de exploración (5 min, en clase)

Es individual: abre el navegador y marca lo que vayas completando:

  • Entra en huggingface.co/inference/models y elige un modelo de chat
  • Abre su ficha y localiza la pestaña Deploy
  • Comprueba si el modelo es gated o de acceso libre
  • Entra en huggingface.co/settings/tokens y localiza dónde se crea un token
  • Busca en huggingface.co/models con el filtro Task → Text Generation

Hugging Face — modelos en local

Sin API, sin cuota: el modelo se ejecuta en tu propia máquina.

Aquí no llamamos a ningún servidor externo: descargamos un modelo open source del Hub y lo ejecutamos con la librería transformers en tu CPU (o GPU, si tienes).

Tu notebook → transformers + PyTorch → modelo en RAM/VRAM → texto generado
HF nubeHF local (hoy)
Libreríahuggingface_hubtransformers + torch
TokenSí (HF_TOKEN)No (modelo público)
DescargaNo, solo APISí (~1 GB la primera vez)
VelocidadDepende del proveedorCPU: lenta · GPU: más rápida
Cuándo usarloProducción, demos rápidasPrototipar sin API ni cuota
Cuantización (si veis modelos "4-bit" o "GGUF"): es como comprimir una foto. El archivo ocupa mucho menos espacio y sigue viéndose bien, pero pierde algo de detalle. Un modelo cuantizado ocupa menos RAM a cambio de algo de precisión. No la usamos hoy, pero la veréis en modelos más grandes.
¿Por qué no usar siempre local? Un modelo grande puede tardar minutos en descargarse y ocupar mucha RAM al cargarlo; en CPU la generación también es más lenta. Por eso hoy usamos un modelo pequeño (0,5B parámetros).

¿Dónde encontrar modelos para local?

1

Buscador del Hub

huggingface.co/models — filtro Task: Text Generation, busca Instruct, Chat o it en el nombre.

2

Revisa el tamaño

Pestaña Files and versions de la ficha del modelo → cuánto vais a descargar.

3

El id que usaréis en código

Formato autor/nombre-repo, p. ej. Qwen/Qwen2.5-0.5B-Instruct.

Cómo leer una tabla de comparación de modelos

Cuando comparéis modelos en herramientas como OpenRouter, Artificial Analysis o cualquier leaderboard, os vais a encontrar tablas con columnas como estas. Saber leerlas es justo lo que necesitáis para la Parte 4 (Benchmark) del Team Challenge.

🏷️ Model

Identificador del modelo, normalmente autor/nombre. El mismo modelo puede aparecer servido por varios proveedores distintos.

🏢 Provider

Quién sirve ese modelo en concreto. Ojo: el mismo modelo en dos proveedores puede tener precio y velocidad muy distintos.

💵 Input $/1M

Coste por cada millón de tokens que vosotros mandáis (prompt + contexto + historial). Suele ser más barato que el output.

💰 Output $/1M

Coste por cada millón de tokens que el modelo genera. Casi siempre más caro que el input — pesa más en respuestas largas.

📏 Context

Ventana de contexto máxima, en tokens: cuánto prompt + historial + respuesta cabe en una sola llamada.

⏱️ Latency(s)

Tiempo hasta que empieza a llegar respuesta (time-to-first-token). Importa mucho en chat en tiempo real.

⚡ Throughput(t/s)

Velocidad de generación una vez ha empezado a responder, en tokens por segundo. Importa en respuestas largas.

🛠️ Tools

Si soporta function/tool calling — el modelo puede pedir ejecutar una función vuestra. Imprescindible para agentes.

📐 Structured

Si soporta forzar el formato de salida (JSON/schema). Clave para integrar la respuesta directamente en vuestro código sin parsear texto libre.
