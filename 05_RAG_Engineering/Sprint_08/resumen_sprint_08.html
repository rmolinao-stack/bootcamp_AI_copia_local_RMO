Idea que atraviesa el módulo Un sistema RAG no es un prompt largo: es un pipeline por capas. Se prepara (S8), se recupera bien (S9) y se genera con robustez (S10). Separar responsabilidades permite depurar por etapa, no culpar al LLM en genérico.
1

Qué es RAG y por qué

Un LLM no conoce por defecto tus documentos privados y tiene una fecha de corte. RAG deja que el modelo responda anclado a tus documentos, sin reentrenarlo cada vez que cambia un PDF.

Las limitaciones que RAG resuelve

Limitación del LLM soloQué significa en la práctica
Conocimiento desactualizadoNo sabe precios, políticas o normas publicadas tras su entrenamiento
Sin acceso a docs privadosNo ha leído tus contratos, manuales ni bases de datos propias
AlucinacionesInventa datos plausibles (fechas, cifras) si no tiene fuente
Ventana finitaNo puedes pegar 500 PDFs enteros en un solo prompt
Poca trazabilidadDifícil saber de qué documento salió una afirmación

RAG en tres palabras

R · Retrieval

Recuperar

Buscar los fragmentos de texto relevantes en tu base de conocimiento (índice vectorial).

A · Augmented

Aumentar

Añadir esos fragmentos al prompt del LLM como contexto.

G · Generation

Generar

El LLM redacta la respuesta usando ese contexto. No memoriza tus PDFs: los consulta en tiempo de inferencia.

Tres formas de dar conocimiento al modelo

EnfoqueIdeaVentajaLímite
Prompt engineeringMetes los datos en el prompt (FAQ, extractos)Rápido, barato, control totalNo escala con corpus grande
RAGRecuperas chunks relevantes y los inyectasEscala, actualizable, trazableDepende del chunking y el retrieval
Fine-tuningReentrenas los pesos con datos etiquetadosComportamiento muy especializadoCaro, lento; no vale para docs que cambian
Cuándo RAG NO es la primera opción 5–10 FAQs fijas, cambiar solo el tono, o un corpus de 1 página → basta prompt manual o pegar el texto. Regla: si el corpus crece, cambia a menudo o no cabe en el prompt → evalúa RAG.

Ingesta: del documento al texto

La ingesta lee cada archivo del corpus y lo convierte en documentos unificados (texto + metadata), sin importar si el origen fue PDF, Markdown, TXT o CSV.

📥 Carga (load)

Loaders de LangChain: PyPDFLoader para PDF, TextLoader para .txt/.md. Un CSV de eventos se convierte fila a fila: una fila = un documento. El loader abstrae el formato: el resto del pipeline no sabe de dónde vino el texto.

🧹 Limpieza (clean)

Quita ruido antes de trocear: saltos de línea raros, espacios de más, líneas vacías repetidas. Un texto uniforme produce chunks más homogéneos. Limpia antes de chunkear, no después (si no, duplicas ruido en cada fragmento).

Metadatos: viajan con cada chunk hasta el vector

Cada fragmento lleva metadatos para saber de dónde vino cuando el retrieval lo devuelva (S9) o cuando cites fuentes (S10).

CampoUso
sourceRuta o nombre del archivo origen
pageNúmero de página (PDFs)
chunk_indexOrden del chunk en el pipeline (0, 1, 2…)
chunk_sizeCaracteres del fragmento (debug)
distrito · tipoMetadatos de dominio (eventos del CSV) para filtrar en retrieval
Por qué importa La metadata distrito, id_evento, tipo sigue viva en embeddings.json: sirve para filtrar o citar fuentes más adelante.
4

Chunking: el factor que más pesa

Chunking = dividir documentos largos en fragmentos pequeños antes de embeddear. Es uno de los factores que más influye en la calidad del RAG: un retrieval malo suele empezar por chunks mal dimensionados.

Por qué no indexar el documento entero

EnfoqueProblema
Documento completo en un embeddingMezcla muchos temas → búsqueda imprecisa
Documento entero en el promptNo cabe en contexto; caro
Chunks pequeños y homogéneosRecuperas el párrafo relevante

La herramienta por defecto: RecursiveCharacterTextSplitter

Intenta cortar respetando la estructura, en este orden de separadores — primero párrafos, luego frases, y solo al final parte una palabra:

"\n\n"  →  "\n"  →  ". "  →  " "  →  carácter
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=100,
    length_function=len,
)
chunks = splitter.split_documents(documentos)

Los dos parámetros clave

chunk_size — tamaño del fragmento

  • Pequeño (300–500): preciso en preguntas concretas, pero pierde contexto de párrafo
  • Medio (700–1000): buen equilibrio general ✅
  • Grande (2000+): más contexto, pero mezcla temas → embedding difuso

chunk_overlap — solapamiento

10–20% del chunk_size. Hace que fragmentos vecinos compartan texto para no cortar una idea a la mitad.

Chunk 1: "...Lugar: Parque de Villa"
Chunk 2: "Parque de Villa Rosa. Distrito..."
         ^^^^^ overlap ^^^^^
Errores frecuentes: chunk_size enorme (5000+) → un chunk = muchas páginas · overlap = 0 → se pierde info en el límite · chunkear antes de limpiar → duplicas ruido · no revisar muestras → el problema aparece recién en S9.
Sin valor universal No hay un chunk_size perfecto: depende del documento. Empieza en ~800 / overlap ~100, inspecciona muestras a mano y ajústalo evaluando el retrieval en el Sprint 9.
5

Embeddings y similitud semántica

Un embedding transforma texto en una lista de números (vector). No se leen los números uno a uno: se usa la distancia entre vectores como proxy de similitud de significado.

De palabras a números

"¿Eventos culturales gratis en Madrid?"
        │
        ▼
[0.02, -0.15, 0.08, ..., 0.11]
   ← 3072 dimensiones (gemini-embedding-2)

Textos semánticamente cercanos → vectores cercanos en el espacio. Importa la posición relativa, no cada componente.

Similitud del coseno

Mide el ángulo entre dos vectores (no su longitud). En texto suele estar entre 0 y 1:

sim = 1.0  → mismo significado
sim = 0.8  → muy parecido
sim = 0.3  → poco relacionado
sim ≈ 0    → temas distintos

En RAG: comparas la pregunta con cada chunk y te quedas con los top-K más altos.

Qué NO es un embedding

ConfusiónRealidad
"Es un hash del texto"Textos distintos con el mismo significado → vectores similares
"Mayor valor = más importante"Importa la dirección del vector, no un componente suelto
"Sirve para generar respuestas"El embedding busca; el LLM redacta después
Regla de oro del RAG Mismo modelo de embedding para indexar los chunks y para embeddear la consulta del usuario. Mezclar Gemini al indexar y Hugging Face al consultar (o modelos distintos) invalida la comparación de distancias.

Comparar modelos de embedding (estilo Sprint 7)

Igual que comparaste LLMs con latencia y calidad, aquí comparas embeddings con pares de frases controlados: sinónimos deben dar similitud alta; temas distintos, baja.

Par de frasesExpectativa
"Live Review del bootcamp" ↔ "Sesión en vivo del sprint"Similitud alta
"¿Cuándo entregar la práctica?" ↔ "Plazo de entrega antes de la Live Review"Similitud alta
"Live Review del bootcamp" ↔ "Receta de tortilla española"Similitud baja
Criterio de elección No elijas un embedding porque suena bien; elige el que separa bien tus pares de prueba en tu dominio. En el bootcamp: Gemini como default (coherencia con el stack), HF para experimento local.

El pipeline modular clave del sprint

Aquí es donde cuesta ver el conjunto. La idea es la misma de siempre: en vez de un script gigante, el pipeline se parte en archivos pequeños, cada uno con una sola responsabilidad, que se pasan el resultado en cadena. Vamos despacio.

La idea en una frase Un archivo = una etapa del pipeline. load lee, clean normaliza, chunk trocea, embed vectoriza. Cada uno recibe lo del anterior y entrega al siguiente. Nadie hace el trabajo de otro.

¿Por qué molestarse? (el problema que resuelve)

❌ Todo en un solo archivo
Un script.py que lee PDFs, limpia texto, trocea, llama a la API de embeddings y guarda JSON, todo mezclado. Si el PDF sale ilegible, no sabes si falló la carga o la limpieza. Cambiar el chunk_size obliga a rebuscar. No puedes reusar la carga en el Sprint 9.
✅ Un archivo por etapa
load.py lee, clean.py limpia, chunk.py trocea, embed.py vectoriza y config.py concentra los parámetros. Depuras por etapa, cambias el tamaño de chunk en una línea y extiendes el proyecto en S9/S10 sin reescribir la ingesta.

El pipeline por capas (de arriba a abajo)

Léelo como un flujo: arriba lo que arranca y decide, en medio las etapas de datos, abajo lo que entra y sale. Cada capa solo habla con la siguiente.

🚪 main.py punto de entrada
El director de orquesta. No hace el trabajo: coordina. Llama a ejecutar_ingesta() y luego a ejecutar_embeddings(), e imprime un resumen. Nada más.
▼ usa
⚙️ config.py configuración central
La "sala de mandos". Aquí y solo aquí viven CHUNK_SIZE, CHUNK_OVERLAP, EMBEDDING_MODEL, MAX_CHUNKS_EMBED y las rutas a data/ y output/. Experimentar = editar una línea aquí, sin tocar la lógica.
▼ alimenta la ingesta
🧠 pipeline.py orquesta la ingesta
Encadena las tres etapas de datos y guarda chunks.json: load → clean → chunk → JSON. También calcula estadísticas (documentos por fuente, chunks por fuente).
▼ llama en orden a
📥 load.py → 🧹 clean.py → ✂️ chunk.py etapas de datos
load: lee .txt/.md/.pdf/.csv como Documents. clean: normaliza el texto. chunk: trocea con el splitter y añade chunk_index y chunk_size. Cada uno recibe una lista de Documents y devuelve otra.
▼ luego
🔢 embed.py + 🔑 gemini_auth.py vectorización
auth: carga la GEMINI_API_KEY una sola vez. embed: lee chunks.json, pide un vector por chunk a Gemini (en lotes) y guarda embeddings.json (texto + vector + metadata).
▼ escribe en
📂 data/ (entrada) · output/ (generado) disco
data/ = corpus de entrada (versionado). output/ = chunks.json y embeddings.json generados en cada run (no se versiona en Git).

Qué hace exactamente cada archivo

config.py
Parámetros y rutas. CHUNK_SIZE, CHUNK_OVERLAP, EMBEDDING_MODEL, MAX_CHUNKS_EMBED, DATA_DIR, OUTPUT_DIR. Fuente única de verdad.
load.py
Carga. Lee .txt, .md, .pdf y .csv desde data/ y devuelve Documents. Convierte cada fila del CSV de eventos en un documento legible.
clean.py
Limpieza. Normaliza el texto (espacios, saltos de línea) y descarta documentos vacíos. Conserva la metadata.
chunk.py
Chunking. Trocea con RecursiveCharacterTextSplitter y enriquece la metadata con chunk_index y chunk_size.
pipeline.py
Orquesta la ingesta. load → clean → chunk, calcula estadísticas y guarda chunks.json.
embed.py
Embeddings. Lee chunks.json, pide vectores a Gemini por lotes y guarda embeddings.json (texto + vector + metadata).
gemini_auth.py
Autenticación. Carga la API key desde .env (o la pide oculta), una sola vez por sesión.
main.py
Orquestador. Ejecuta ingesta + embeddings y muestra un resumen. No contiene lógica propia: solo llama a los demás.

Cómo se hablan los módulos: los import

main.py y pipeline.py son casi solo imports y llamadas — reparten el trabajo, no lo hacen:

# main.py — reparte el trabajo, no lo hace
from pipeline import ejecutar_ingesta
from embed import ejecutar_embeddings

ejecutar_ingesta()        # data/ → chunks.json   (load+clean+chunk)
ejecutar_embeddings()     # chunks.json → embeddings.json  (Gemini)

# pipeline.py — encadena las etapas de datos
from load  import cargar_documentos
from clean import limpiar_documentos
from chunk import fragmentar_documentos

crudos  = cargar_documentos()        # load.py
limpios = limpiar_documentos(crudos) # clean.py
chunks  = fragmentar_documentos(limpios) # chunk.py

Y las dependencias fluyen en una sola dirección (nadie forma un círculo):

config.py          # no depende de nadie — solo parámetros
   ▲
load → clean → chunk        # etapas de datos, en cadena
             ▲
        pipeline.py         # orquesta la ingesta → chunks.json
             ▲
gemini_auth → embed.py      # chunks.json → embeddings.json
             ▲
          main.py           # arriba del todo: solo coordina
Cómo explicarlo con tus palabras Imagina una fábrica de conservas: config es la hoja de especificaciones (tamaño de lata, etiqueta), load recibe la materia prima (PDF, CSV), clean la lava, chunk la corta en porciones iguales, embed le pone el código de barras (el vector) y main es el capataz que pone en marcha la cadena. Cada máquina hace una cosa y pasa el producto a la siguiente.

Qué ganas al modularizar así

🔧 Experimentas fácil

Cambiar CHUNK_SIZE o el modelo = una línea en config.py. No tocas load ni embed.

🐛 Depuras por etapa

¿PDF ilegible? Es load. ¿Chunks partidos? Es chunk. No culpas al pipeline entero.

♻️ Reutilizas y extiendes

En S9 añades index.py y retriever.py; en S10, prompts y logic. La ingesta no se reescribe.

📄 Artefactos inspectables

chunks.json se lee a mano: detectas errores antes de gastar API en embeddear.

💾 Iteras sin recargar

embed lee chunks.json: pruebas embeddings sin re-ejecutar load/chunk.

👥 Trabajo en equipo

Cada persona toca un archivo distinto sin pisarse.

El proyecto: ingesta + chunking + embedding

Pipeline offline completo en un comando: python main.py. Corpus de ejemplo: agenda cultural de Madrid (FAQ .md, guía .txt, PDF oficial y CSV de eventos).

Puesta en marcha

python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # GEMINI_API_KEY
python main.py

Por defecto MAX_CHUNKS_EMBED = 5 (demo rápida). Pon None para embeddear todo el corpus.

Qué genera

  • output/chunks.json — fragmentos + metadata
  • output/embeddings.json — texto + vector (3072 dims) + metadata
  • Un resumen en consola: documentos, chunks por fuente, muestra de vector

Orden recomendado para explorar el código

1
data/ — qué documentos entran al pipeline.
2
config.py — CHUNK_SIZE, CHUNK_OVERLAP, EMBEDDING_MODEL, MAX_CHUNKS_EMBED.
3
Ingesta: load.py → clean.py → chunk.py → pipeline.py.
4
Embeddings: embed.py → output/embeddings.json.
5
Tras ejecutar: inspecciona chunks.json y embeddings.json — abre un item en ambos y comprueba que el texto y la metadata coinciden.

Conclusiones que deberías poder explicar

Varias fuentes, un pipeline

El mismo flujo trata Markdown, PDF y CSV de eventos.

El chunk es la unidad de búsqueda

En retrieval no se recuperan archivos enteros, sino fragmentos.

La metadata viaja hasta el vector

distrito, id_evento, tipo siguen en embeddings.json.

Un modelo, un espacio vectorial

La pregunta del usuario deberá embeddearse con el mismo modelo al buscar.

Si al terminar echas en falta "hacer una pregunta y ver respuesta"… …es señal de que entendiste el pipeline: falta la capa de búsqueda (S9) y de generación (S10), no que algo esté roto. Con embeddings.json el corpus está listo para indexar.

Checklist del sprint

Conceptos que debes dominar

  • Qué limita a un LLM solo y cuándo RAG aporta valor
  • Pipeline offline (indexar) vs online (consultar)
  • Recuperación ≠ generación
  • chunk_size / overlap y su efecto en el retrieval
  • Embedding = texto→vector; similitud del coseno
